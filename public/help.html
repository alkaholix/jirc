jIRC — Help & Guide Using the client, and the full mIRC-style scripting reference

jIRC is a modern, cross-platform IRC client that speaks standard IRC (RFC 1459/2812 + IRCv3) and IRCX, with a built-in mIRC-style scripting engine. The first part of this guide covers using the app; the second part is the complete scripting reference.

Connecting to a server

  • Click + (or "Connect") to open the connect dialog. Enter the server host and port, tick TLS for a secure connection, set your nick (and an alternative used if it's taken), and any channels to auto-join.
  • Saved servers appear in a dropdown — pick one and connect, or delete it.
  • You can connect to several servers at once; each gets its own console and windows, and auto-reconnects if dropped.
  • Passwords (server / NickServ) are stored in your OS keyring, not in plain text.

The interface

  • Layout: choose a collapsible tree sidebar or a switchbar of tabs in Settings → Appearance.
  • Each server has a console (status) window; channels and private chats (queries) open as their own windows.
  • The nick list (right side of a channel) shows members with their rank prefixes; the topic bar shows/edits the topic.
  • Buttons top-left: ⟨⟩ scripts, ⚙ settings, + add a connection.

Chatting & commands

Type in the input box and press Enter to talk. Lines starting with / are commands. Common ones:

CommandDoes
/join #chan · /partjoin / leave a channel
/msg nick text · /query nickprivate message / open a query
/me actionsend an action
/nick newnickchange your nick
/whois nicklook someone up
/topic [text]view / set the channel topic
/list · /listxbrowse the channel list (IRCX /listx)
/away [msg]set / clear away (or use the Away button)
/ignore nick · /unignorehide / unhide someone
/raw linesend a raw IRC line

Anything jIRC doesn't recognise is passed to your scripts (an alias) and then to the server, so your own /commands work here too.

Formatting, colours & emoji

  • Incoming mIRC colour/format codes render in full; URLs are clickable.
  • Type :shortcodes: like :smile: or :fire: — they expand to emoji.
  • Add your own in Settings → Appearance → Custom emoji: map a :code: to text/unicode (sent as text) or an image URL (shown inline as a little picture).

Find in buffer

Press Ctrl/Cmd+F to search the current window. Matches are highlighted with a count; Enter / Shift+Enter jump to the next / previous; Esc closes.

People, presence & the notify list

  • Right-click a nick for whois / query / whisper / op / voice / kick / ban / ignore (this menu is script-customisable, see Popups).
  • Away: the topic bar's Away button sets/clears your away state (with a message) and lights up while you're away.
  • Notify / watch list: add friends' nicks in Settings → Alerts. jIRC checks who's online and alerts you when they come/go, and lists who's around in the sidebar's Online panel.
  • Highlight words & ignores are also in Settings → Alerts.

Channel management

  • Op/voice/kick/ban from the nick-list menu, or with /mode, /kick, /ban.
  • Double-click the topic bar to edit the topic.
  • /list opens a sortable, filterable channel browser (IRCX servers: /listx).

Customising jIRC

  • Theme: dark / light / match-system (Settings → Appearance).
  • Your nick colour is configurable; other nicks use a standard colour.
  • Custom emoji (text or inline images) — see above.
  • Nicklist icons: scripts can badge a nick with /nickicon nick 👑 (an emoji or image URL).
  • Custom CSS: Settings → Appearance has a CSS box (with a built-in cheat sheet) to restyle anything — colours, fonts, spacing, hide elements — applied live. There's a Reset button if you go too far.
  • Custom dialogs & popups let scripts build their own UI — see the scripting section.

Custom CSS cheat sheet

The whole app is drawn with CSS, so you can restyle anything from Settings → Appearance → Custom CSS. CSS is just selector { property: value; } — pick something to change, set a property, done. The same cheat sheet is built into the settings panel. Copy an example and tweak it:

/* recolour the whole app */
:root { --accent: #ff4da6; }

/* bigger chat font */
.messages { font-size: 16px; }

/* hide timestamps */
.line .time { display: none; }

/* compact member list */
.nick-entry { padding: 1px 4px; }

/* a background image */
.messages { background: url("https://…/bg.png") center/cover; }

Security & where your data lives

  • TLS (rustls), SASL PLAIN, NickServ identify, and a SOCKS5 proxy are all supported.
  • Passwords live in the OS keyring (Windows Credential Manager / macOS Keychain / Linux Secret Service), never in plain files.
  • Server profiles: profiles.json; scripts: scripts/*.mrc; chat logs: logs/<network>/<buffer>.log — all under the app config/data dir.

Scripting (mSL)

A substantial subset of mIRC's scripting language: your own /commands (aliases), automatic reactions (events), right-click menus (popups), custom dialogs, timers, regex, file I/O and sockets.

This is a subset of mIRC — not 100% compatible. See What's not supported for the boundaries.

Introduction

A script is plain text made of aliases, events, and menus. Each lives in a .mrc file. All your .mrc files are compiled together, so an alias in one file can be called from another.

Where scripts live & how to load them

  • Open the Scripts dialog in jIRC (the ⟨⟩ button).
  • Create/edit files there, or click Add examples to drop in starter scripts.
  • On disk they're under the app config folder:
    • Windows: %APPDATA%\com.jirc.app\scripts\
    • macOS: ~/Library/Application Support/com.jirc.app/scripts/
    • Linux: ~/.config/com.jirc.app/scripts/
  • Saving in the Scripts dialog recompiles everything immediately — no restart needed.

Quick start

Paste this into a script file and save. Then type /hello in any channel, and watch what happens when someone joins.

; my first script
alias hello { /msg $chan Hello from a script, $me$+! }

on *:JOIN:#:{ /msg $chan Welcome $nick to $chan }

Scripted messages are echoed into your own window too, just like in mIRC, so you'll see the output yourself.

Syntax & comments

  • A line starting with ; is a comment.
  • Statements are separated by a new line or a pipe |.
  • Blocks are wrapped in { }.
  • $identifiers return values; %variables are yours to set.
  • $+ joins two pieces together with no space: $me $+ ! → YourNick!
; two statements on one line
alias greet { /echo hi | /echo there }

Aliases (your own /commands)

An alias defines a command you can type. Use $1, $2, … for the words the user typed after it.

alias slap { /me slaps $1 around a bit with a large trout }
alias op   { /mode $chan +o $1 }
alias shrug { /say ¯\_(ツ)_/¯ }

Type /slap bob → sends an action "slaps bob …".

Parameters: $1, $2, $2-, $0

TokenMeaning
$1, $2 …The first, second … word.
$2-The second word onward (the rest of the line).
$0How many words there are.
alias kickreason { /kick $chan $1 $2- }
; /kickreason bob being rude  ->  KICK #chan bob :being rude

Variables

Variables start with %. They persist for the session.

CommandEffect
/set %x valueSet a variable.
/inc %x [n]Increase by n (default 1).
/dec %x [n]Decrease by n.
/unset %xRemove it.
alias count {
  /set %i 0
  /while (%i < 3) { /inc %i | /echo tick %i }
}

Dynamic variables

You can build a variable name on the fly with %name. [ $+ [ <value> ] ] — the part in brackets is worked out first and stuck onto the name. Handy for per-person data or array-style lists:

on *:TEXT:!setcolor &:#:{ /set %color. $+ $nick $2 }
on *:TEXT:!mycolor:#:{ /msg $chan $nick $+ 's colour is %color. [ $+ [ $nick ] ] }

The $eval($+(%color.,$nick),2) form does the same thing if you prefer it.

Custom identifiers (return a value)

Any alias can be used as an $identifier if it returns a value with /return. This lets you build reusable helpers.

alias double { /return $calc($1 * 2) }

alias test { /echo twice five is $double(5) }
; -> twice five is 10

Control flow

if / elseif / else and while loops. Bodies can be a { } block or a single brace-less statement on the same line (it runs to the end of the line or the next |).

alias grade {
  if ($1 >= 90) { /echo A }
  elseif ($1 >= 80) { /echo B }
  else { /echo keep trying }
}
; brace-less form — common in mIRC scripts:
alias check {
  if ($1 == $null) return
  if ($1 isnum) /echo got a number | /echo always runs
}

goto / labels

A line :name is a label; /goto name jumps to it (it can jump out of if/while blocks). A safety cap stops runaway loops.

alias countup {
  /set %i 0
  :loop
  /inc %i
  /echo %i
  if (%i < 5) { /goto loop }
  /echo done
}

Events: reacting with on

Events run automatically. The general form is:

on <level>:<EVENT>:<target>:{ commands }

The level is usually * (it's accepted but not enforced). Text-style events also take a matchtext pattern before the target:

on *:TEXT:<matchtext>:<target>:{ commands }

The matchtext is a wildcard pattern: * matches anything (including nothing), ? matches one character, and & on its own matches exactly one word. So on *:TEXT:!weather &:#: triggers on !weather london (with $2 = london) but not on !weather alone.

The target field

TargetMatches
#any channel
#channelthat specific channel
?a private/query message (no channel)
*anywhere

Examples

; reply to a trigger word
on *:TEXT:!ping*:#:{ /msg $chan pong $nick }

; greet joiners
on *:JOIN:#:{ /msg $chan welcome $nick }

; announce kicks
on *:KICK:#:{ /msg $chan $knick was kicked by $nick ($1-) }

; log mode changes
on *:MODE:#:{ /echo -s $nick set $1- on $chan }

; react to your OWN input
on *:INPUT:#:{ if ($1 == !time) { /msg $chan it is $time } }
on TEXT fires for other people's messages, not your own — use on INPUT for your own typed lines. An on INPUT handler can call /halt to stop your line from being sent (e.g. to intercept a custom command).

Script groups

Group related aliases and events so you can switch them on or off together. A group is a range opened by #name on (or #name off to start disabled) and closed by #name end. Aliases and events inside a disabled group don't fire.

#greeter on
on *:JOIN:#:{ /msg $chan welcome $nick }
alias hi { /msg $chan hi all }
#greeter end

Toggle groups at runtime — names may use wildcards (#help*, or #* for all):

/disable #greeter      ; turn it off
/enable #greeter       ; turn it back on
/groups               ; list groups (-e enabled only, -d disabled only)

$group(0) is the number of groups, $group(N) the Nth name, and $group(#name).status is on or off.

Signals — talk between scripts

/signal <name> [params] fires every on SIGNAL handler whose name matches (wildcards allowed). In the handler, $signal is the name and $1- are the parameters.

on *:SIGNAL:refresh:{ /echo -s refreshing ($1-) }
alias go { /signal refresh now }

More commands

  • /alias <name> <commands> defines a single-line alias at runtime; /alias <name> with no commands removes it.
  • /anick, /mnick, /fullname set your alternate nick, main nick and full name (read back as $anick/$mnick/$fullname).
  • /autojoin -n|-s|-dN — use in on CONNECT to join now, skip, or delay the connect-time autojoin by N seconds. (Manage the channels themselves in the # auto-join dialog.)
  • /reload recompiles all script files; /unsetall clears every variable.

Popup menus

Add items to the right-click menu in the nick list. Leading dots make submenus; a line that is just - is a separator. In the nick-list menu, $1 is the nick you clicked.

menu nicklist {
  Whois:/whois $1
  -
  Control
  .Op:/mode $chan +o $1
  .Deop:/mode $chan -o $1
  .Kick:/kick $chan $1
  -
  Slap:/me slaps $1 around a bit
}

Selected nicks, styles & dynamic items

In a popup…Gives you
$snicksthe selected nicks as nick1,nick2,…
$snick(#, N)the Nth selected nick ($snick(#, 0) = count)
$style(N)as an item's first word: 1 adds a check mark, 2 greys it out (disabled), 3 both
$submenu($id($1))build items dynamically — calls your $id with $1 = begin, then 1, 2, … until it returns nothing, then end
menu nicklist {
  ; grey out "Whois" when you click your own name
  $iif($1 == $me,$style(2)) Whois $1:/whois $1
  -
  Colour the channel
  .$submenu($colour($1))
}
alias colour {
  if ($1 == 1) /return Red:/msg $chan red
  if ($1 == 2) /return Blue:/msg $chan blue
}

Custom dialogs

Build your own pop-up windows with input fields, dropdowns and buttons. Unlike mIRC, there are no pixel coordinates — controls are given a name and laid out automatically, and they render as real, themed web controls.

Defining a dialog

dialog quicksay {
  title  "Quick say"
  text   info  "Type a message:"
  edit   msg
  combo  where "#test" "#lobby"
  check  act   "Send as an action"
  button send  "Send" :default
  button cancel "Cancel" :cancel
}
ControlRenders as
text id "label"a static label
edit id ["initial"]a single-line input
editbox ida multi-line text area
check id "label"a checkbox ($did = 1/0)
combo id "opt"…a dropdown
list id "opt"…a list box
button id "label" [:default] [:cancel]a button

Opening, reading, reacting

  • /dialog name opens it; /dialog -c name closes it.
  • $did(name, control) reads a control's current value.
  • /did [-a|-r] name control [text] changes a control: -a add a combo/list item, -r clear, default set the value.
  • on *:DIALOG:name:{ } fires on interaction; $1 is the control that triggered it (a button id, or init when it opens).
alias qsay { /dialog quicksay }

on *:DIALOG:quicksay:{
  if ($1 == send) {
    if ($did(quicksay, act) == 1) { /describe $did(quicksay, where) $did(quicksay, msg) }
    else { /msg $did(quicksay, where) $did(quicksay, msg) }
    /dialog -c quicksay
  }
}
Behind the scenes, the dialog's control values are sent to your script each time you interact, so $did always reads the current form state. Use on *:DIALOG:name: with $1 == init to pre-fill controls with /did when it opens.

Timers

/timer[name] <reps> <seconds> <command> runs a command repeatedly. reps of 0 means "keep repeating". Give a name to stop it later.

CommandEffect
/timer reps secs cmdstart an (auto-named) timer
/timerNAME reps secs cmdstart a named timer (replaces same name)
/timerNAME offstop that timer
/timers · /timers offlist all · stop all
; say "tick" 5 times, once per second
alias ticker { /timer 5 1 /msg $chan tick }
; a named, repeating announcer you can stop with /stopann
alias ann    { /timerann 0 300 /msg $chan visit our site! }
alias stopann { /timerann off }

alias banhost { /mode $chan +b $address($1, 3) }
; /banhost bob  ->  MODE #chan +b *!*@bob's.host

; voice everyone in the current channel
alias voiceall {
  /set %i 1
  :loop
  if (%i <= $nick($chan, 0)) {
    /mode $chan +v $nick($chan, %i)
    /inc %i | /goto loop
  }
}

Token lists

alias demo {
  /echo $gettok(a.b.c, 2, 46)      ; -> b
  /echo $sorttok(c a b, 32)         ; -> a b c
  /echo $addtok(red green, blue, 32) ; -> red green blue
}

Regular expressions

IdentifierReturns
$regex(text, pattern)number of matches; stores capture groups
$regml(n)the nth captured group from the last $regex
$regsub(text, pattern, repl)text with replacements (\1 backrefs)
alias parse {
  if ($regex($1-, ([a-z]+)(\d+))) {
    /echo letters= $regml(1)  digits= $regml(2)
  }
}
; /parse abc123  ->  letters=abc  digits=123

alias censor { /return $regsub($1-, \d, #) }
; $censor(call 5551234) -> call #######
Uses standard (PCRE-like) regex syntax via the Rust regex engine. The optional named-context first argument of mIRC's $regex is accepted but ignored.

Files sandboxed

Scripts can read and write small text files. For safety, file access is sandboxed: only the file name is used, and all files live in a dedicated folder (scriptdata next to your scripts). Scripts can't read or write anywhere else on your disk.

Sockets (sockbots) advanced

Scripts can open raw TCP connections — enough to build sockbots, talk to other services, or speak a custom protocol. Sockets are line-oriented; use /sockopen -e for a TLS (encrypted) connection.

Security: sockets let a script connect anywhere on the network — they intentionally step outside the file sandbox. Only run socket scripts you trust.
Command / Identifier / EventEffect
/sockopen [-e] name host portopen a TCP socket (-e = TLS)
/sockwrite [-n] name textsend text; -n appends CRLF
/sockclose nameclose the socket
/socklist [wild]list open sockets (optional wildcard)
/sockread %var(in on SOCKREAD) read the current line
on *:SOCKOPEN:name:{ }fired when connected
on *:SOCKREAD:name:{ }fired per incoming line (also in $1-)
on *:SOCKCLOSE:name:{ }fired when the socket closes
$socknamethe socket's name
$sockbrbytes read by the last /sockread

The socket name in an event (e.g. :mybot:) is matched as a wildcard, and socket events run in the context of the connection that opened them — so /msg #chan from a handler goes to that network.

; a minimal bot that connects to an IRC server over a socket
alias startbot { /sockopen mybot irc.example.org 6667 }

on *:SOCKOPEN:mybot:{
  /sockwrite -n mybot NICK jbot
  /sockwrite -n mybot USER jbot 0 * :jbot
}

on *:SOCKREAD:mybot:{
  var %line
  /sockread %line
  ; reply to the server's PING so we stay connected
  if ($1 == PING) { /sockwrite -n mybot PONG $2- }
}

on *:SOCKCLOSE:mybot:{ /echo -s mybot disconnected }

The simple form works too — the line is in $1-, so you can skip /sockread entirely: on *:SOCKREAD:mybot:{ if (PING isin $1-) { … } }.

Safety: other people's text stays text

This is one place jIRC deliberately behaves differently from mIRC — and it's in your favour. In mIRC, a few things get evaluated twice, which means a stranger's message (or a line in a file) can quietly turn into commands your client then runs. That's the source of many "my bot got hijacked" stories. jIRC evaluates once and treats data as data, so those tricks simply don't work here.

Say you run a reminder bot:

on *:TEXT:!remind *:#:{ /timer 1 $2 notice $nick $3- }

Now someone types:

!remind 5 hi there | quit hacked!

In mIRC, when the timer fires 5 seconds later it re-reads its command, sees the | as a command separator, and actually runs /quit hacked! — the attacker just quit your client. In jIRC, the reminder text is treated as plain text, so your bot just says "hi there | quit hacked!" and nothing bad happens.

The same protection applies elsewhere:

  • Reading files ($read, $readini): mIRC treats each line of a file as code; jIRC returns the line as plain text.
  • Maths ($calc): in mIRC $calc(%password) can leak a variable's value; in jIRC $calc only does maths on what you actually give it.
  • Running commands on other connections / after a delay (/timer, /scon, /scid): the command text is kept as-is instead of being re-run through the interpreter.

The upshot: you don't need mIRC's $safe/$unsafe work-arounds in jIRC — text from other people, or from files, stays text.

What's not supported

Intentionally left out (by design)

These will not be added — they're Windows-only and/or unsafe, and go against jIRC being cross-platform and safe to share:
  • COM/DLL bridges ($com, $dll) and DDE.
  • Loading native binaries / arbitrary executables from scripts.

Now supported

Landed since early builds — these do work now: DCC (chat + file send/get), custom @windows, binary variables (&binvar/$bvar//bset), $regsubex, crypto ($md5/$sha1/$hmac), $input, .ini files ($readini//writeini), most .property suffixes ($sock(x).status, $hget(t,N).item), and IRCX owner/host keys. Some aren't in the reference tables above yet.

Not yet (may come later)

Planned or possible future work — scripts using these won't work today:
  • Picture / drawing @windows, and some lifecycle events (on OPEN, on CLOSE, on NOTIFY).
  • A few property suffixes such as $chan(#).topic — most now work, just not every one.
  • A long tail of rarely-used identifiers.

Two gotchas worth remembering:

  • An unknown identifier prints as literal text (e.g. a typo like $madeupname shows up as the words "$madeupname").
  • An unknown client-side command (windows/display/sound) is ignored; any other unknown command is sent to the server as a raw IRC line.
